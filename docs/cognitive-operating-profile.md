# Cognitive Operating Profile — Dave

> How Dave's brain works, what enables peak performance, and what creates friction. Use this to adapt collaboration style, task design, and system architecture.

---

## Peak Performance State

**Core driver:** Forward motion and momentum

Dave's peak performance state is when **ideas gain momentum and build on each other**. This is the zone where productivity compounds.

### What Enables This State

- **Uninterrupted flow** — continuous blocks of time to think, build, and iterate
- **Familiar territory** — working in systems and contexts he knows well
- **Early mornings or late nights** — fewer external disruptions (phone calls, people interactions)
- **High confidence decisions** — 90%+ confidence with clear monitoring thresholds
- **Strong momentum** — when one win leads to the next, creating a cascade of progress

### What Protects This State

- **Minimize context switches** to unfamiliar domains
- **Reduce unaccounted-for tool breakage** — his biggest productivity blocker
- **Front-load setup complexity** so deep work flows without interruption
- **Provide monitoring signals** for high-stakes decisions so course correction is fast
- **Shield from unexpected life events** when possible (these are rare but disruptive)

---

## Context Switching

Dave has **two modes of context switching** — one is easy, one is crippling.

### ✅ Easy Context Switching

**Conditions:**
- Switching between familiar systems he can navigate fluently
- All tasks are "on his radar" — he knows the domain well
- No reading of complex instructions or buried settings required
- He's already memorized the process

**Example:** Jumping between multiple coding tasks in the same codebase, or between familiar tools he uses daily.

### ❌ Crippling Context Switching

**Conditions:**
- New systems with unfamiliar UI or buried settings (3-4 levels deep)
- Reading generic instructions that don't match where he lands
- Outdated instructions that no longer reflect the current UI
- Bringing information from one new context into another
- **Root cause:** High cognitive load from reading + context transfer

**Example:** "Sign up for a new service, find a key buried in settings, follow generic instructions that are outdated or don't match the actual screens."

**Once he's done the system before and memorized it, the difficulty disappears.**

## Momentum & Recovery

### Momentum Protection

- **Size doesn't matter** — large or small tasks are fine
- **Focus matters** — what matters is whether the task maintains flow
- **When momentum is strong, protect it as long as possible** — don't break unless forced to
- **Break only when necessary:**
  - Time constraints (running up against midnight)
  - Physical exhaustion (truly getting tired)

### Derailment Triggers

**Primary trigger:** Unexpected life events that force time reallocation
- Someone in his life needs help
- Surprise situation in another area requiring immediate attention
- **These are not frequent but hit hard when they happen**

**Not triggers:**
- Noise or environmental distractions (he filters these out)
- Normal interruptions

### Recovery from Breaks

**Key decision:** Is the value in restarting and regaining momentum, or walking away and coming back later?

- **This is learned over time** — he's developed instinct for when to push vs. when to stop
- **Forcing it can backfire** — sometimes trying to restart becomes frustrating
- **The right call:** Knowing when to just walk away and come back fresh

## Decision-Making

Dave **gathers data before deciding**, but has two distinct decision modes depending on stakes and clarity.

### Fast Decisions (90%+ Confidence)

**When:**
- He has high confidence (90%+)
- Monitoring thresholds are clear and known
- He understands what to watch for if things go wrong

**Process:**
- Quick evaluation
- Execute immediately
- Fast course correction if needed (because monitoring signals are clear)

### Gated Decisions (Requires Deeper Measurement)

**When:**
- Confidence is lower
- Stakes are high
- Unclear what to monitor or how to detect failure
- Uncertainty about what could go wrong

**Process:**
- Gather more data
- Build understanding of monitoring signals
- Increase confidence before committing
- **Goal:** Not just higher confidence, but higher understanding of what to watch for

**Why this matters:** For high-stakes decisions, Dave prioritizes understanding failure modes and monitoring signals so he can course-correct quickly and preserve momentum.

## Task Management

### What Works

- **Familiar task clusters** — multiple related tasks in a known domain
- **Clear sequences** — tasks with obvious order and dependencies
- **Project grouping** — tasks organized by project with ordering within each group
- **Manual control** — ability to reorder and reprioritize as needed

### What Doesn't Work

- **Generic instructions that don't match reality** — outdated docs, vague steps
- **Buried complexity** — settings or keys hidden multiple levels deep
- **Forced task switching** to unfamiliar domains mid-flow
- **Unaccounted-for tool breakage** — surprises that require hard stops and momentum rebuilding

## Collaboration Style

### How to Support Dave

**Protect momentum:**
- Minimize surprise blockers
- Front-load setup complexity
- Provide clear monitoring signals for decisions
- Shield from unnecessary context switches

**Communicate effectively:**
- Use bullet points and clear structure (dyslexia-friendly)
- Bold key terms for scannability
- Short paragraphs, not walls of text
- Never correct spelling or grammar

**Decision support:**
- For fast decisions: Give clear options with monitoring thresholds
- For gated decisions: Help build understanding of failure modes and signals
- Always prioritize "what to watch for" over "this will definitely work"

**Task design:**
- Keep tasks within familiar domains when possible
- Group by project with clear ordering
- Make setup/complexity explicit upfront
- Provide detailed first-time instructions that match reality

### What to Avoid

- **Never nag** — gentle accountability, not pressure
- **Never force momentum breaks** unless absolutely necessary
- **Never assume instructions are current** — verify or provide screenshots
- **Never create surprise blockers** — surface risks and dependencies early
- **Never treat struggles as character flaws** — normalize executive function challenges

## System Design Implications

### Architecture Principles

1. **Reduce cognitive load for unfamiliar systems**
   - Clear, current instructions
   - Minimal buried settings
   - Obvious paths to common operations

2. **Protect momentum**
   - Minimize unaccounted-for tool breakage
   - Fast error recovery
   - Clear monitoring signals

3. **Enable fast course correction**
   - Observable failure modes
   - Clear thresholds and alerts
   - Rapid feedback loops

4. **Support both decision modes**
   - Fast path for high-confidence decisions
   - Data-gathering path for gated decisions
   - Clear monitoring for both

### Feature Design

- **Accessibility first** — dyslexia-friendly formatting is non-negotiable
- **Explicit over implicit** — make complexity visible, don't hide it
- **Familiar patterns** — reuse known interaction models when possible
- **Recovery-oriented** — design for momentum breaks, not just continuous flow

## Working Hours

**Peak productivity windows:**
- Early mornings
- Late nights

**Why:** Fewer external disruptions like phone calls or people interactions.

**Implication:** Schedule deep work during these windows. Protect them from meetings or collaborative sessions when possible.
